# 001 - 通用小说爬虫架构设计

## 目标

将当前仅支持 hetushu.com 的爬虫改造为通用小说爬虫。用户只需提供一个章节链接，系统自动分析页面结构，完成书名识别、标题提取、内容抓取、翻页爬取，以及数据清洗。

## 现状分析

当前代码中硬编码了 hetushu.com 的页面结构：

| 硬编码项 | 当前值 | 说明 |
|---------|--------|------|
| `BASE_URL` | `https://www.hetushu.com` | 域名拼接用 |
| `START_URL` | 固定 URL | 起始章节 |
| 内容选择器 | `#content` | 正文容器 |
| 大标题选择器 | `h2`（无 class） | 卷名 |
| 小标题选择器 | `h2.h2` | 章节名 |
| 下一页选择器 | `a#next` | 翻页链接 |
| 书名提取 | `title` 标签按 `-` 分割 | 取最后一段 |
| 水印清洗 | `hetushu.com.com`、`-图-书` | clean.js 中的正则 |

## 设计方案：适配器模式 + 自动探测

### 整体架构

```
┌─────────────────────────────────────────────┐
│                  crawler.js                  │
│  (通用爬取引擎，不依赖任何特定站点结构)         │
─────────────────────────────────────────────┤
│              SiteAdapter 层                   │
│  (每个站点一个适配器，封装页面结构差异)          │
├──────────┬──────────┬──────────┬────────────┤
│ Hetushu  │ Biquge   │ 69shu    │ AutoDetect │
│ Adapter  │ Adapter  │ Adapter  │  (兜底)     │
└──────────┴──────────┴──────────┴────────────┘
```

### 核心思路

1. **已知站点**：通过域名匹配预置适配器，100% 准确
2. **未知站点**：通过启发式规则自动探测页面结构，覆盖大部分小说站
3. **用户覆盖**：支持通过配置文件手动指定选择器，兜底自动探测失败的情况

### 已知站点适配器示例

```javascript
// adapters/hetushu.js
class HetushuAdapter extends SiteAdapter {
    match(url) {
        return url.includes('hetushu.com');
    }

    extractNovelName($) {
        const title = $('title').text().trim();
        const m = title.split(/\s*[-–—]\s*/);
        return m.length >= 2 ? m[m.length - 1].trim() : title;
    }

    extractContent($) {
        return $('#content');
    }

    extractTitles($, contentDiv) {
        let mainTitle = '';
        let subTitle = '';
        contentDiv.find('h2').each((_, el) => {
            const $el = $(el);
            const text = $el.text().trim();
            if (!text) return;
            if ($el.attr('class') === 'h2') {
                subTitle = subTitle || text;
            } else if (!mainTitle) {
                mainTitle = text;
            }
        });
        return { mainTitle, subTitle };
    }

    getNextUrl($, currentUrl) {
        const nextLink = $('a#next');
        if (nextLink.length === 0) return null;
        const href = (nextLink.attr('href') || '').trim();
        if (!href || href === '#' || href.startsWith('javascript:')) return null;
        return href.startsWith('http') ? href : 'https://www.hetushu.com' + href;
    }

    getWatermarkPatterns() {
        return [
            /https?:\/\/[^\s，。！？、；：""''（）\n]+/g,
            /[\w-]+\.com(?:\.\w+)?/g,
            /-?图-?书/g,
            /和-?图-?书/g,
            /和杀阵/g,  // 清洗后残留
        ];
    }
}
```

### 自动探测适配器（核心）

当 URL 不匹配任何已知站点时，使用 `AutoDetectAdapter` 通过启发式规则分析页面结构。

#### 1. 书名提取策略（按优先级）

```
优先级 1: <meta property="og:novel:book_name" content="xxx">
优先级 2: <meta name="og:novel:book_name" content="xxx">
优先级 3: 面包屑导航中最后一个链接文本（如：首页 > 玄幻小说 > 吞噬星空）
优先级 4: <title> 标签，按常见分隔符分割后取第一段
          分隔符：-、—、–、|、_、《》
          例："第九章 归来 - 吞噬星空 - 和图书" → "吞噬星空"
优先级 5: 页面中最大的 <h1> 标签文本
```

#### 2. 正文内容区域探测策略

```
策略 1: 查找 id 包含 content/text/article/chapter 的元素
        候选 id: content, nr_content, bookcontent, chaptercontent,
                 text, article, main, readcontent

策略 2: 查找 class 包含 content/text/chapter 的元素
        候选 class: content, nr_content, bookcontent,
                    chapter-content, text-content, readcontent

策略 3: 如果策略 1/2 找到多个候选，选择文本最长的一个

策略 4: 如果都没找到，选择页面中 <p> 标签最密集的父容器
        （遍历所有 div，计算其内部 <p> 标签数量和总文本长度，
         取综合得分最高的）

策略 5: 兜底 — 取 <body> 中 <p> 标签的集合
```

#### 3. 标题提取策略

```
策略 1: 内容区域内的 <h1>、<h2>、<h3> 标签
        - 第一个大标题（h1 或无 class 的 h2）→ 卷名
        - 带 class 的 h2/h3 或第二个标题 → 章节名

策略 2: 内容区域外的标题
        - 查找内容容器兄弟节点或父节点中的 h1/h2

策略 3: 从 <title> 标签提取章节名
        - 例："第九章 归来 - 吞噬星空" → "第九章 归来"

策略 4: 兜底 — 使用文件名或序号作为章节名
```

#### 4. 翻页链接探测策略

```
下一页探测（按优先级）：

优先级 1: id/class 包含 next/下一页 的 <a> 标签
          候选: #next, .next, #next_page, .next_page,
                #nextchapter, a[title*="下一"]

优先级 2: 文本包含 "下一章"、"下一页"、"下一"、">>" 的 <a> 标签

优先级 3: 页面底部区域（最后一个 div 或特定容器）中
          URL 与当前页 URL 模式相似但序号+1 的 <a> 标签
          例：当前 /book/10319/7209353.html
              匹配 /book/10319/7209354.html

优先级 4: 所有 <a> 标签中，href 与当前页同目录、
          文件名数字部分恰好+1 的链接

上一页探测（用于验证探测准确性）：
  同上策略，关键词替换为 prev/上一章/上一/<<
```

#### 5. 编码检测

```
策略 1: 检查 <meta charset="xxx"> 声明
策略 2: 检查 HTTP Content-Type 响应头
策略 3: 检查 BOM 头
策略 4: 使用 iconv-lite 的 detectEncoding 检测
        （部分站点返回 GBK/GB2312 编码）
```

### 配置文件设计

当自动探测失败时，用户可以通过配置文件手动指定选择器：

```json
// site-config.json（可选，放在项目根目录）
{
    "sites": {
        "example.com": {
            "novelName": {
                "selector": "meta[property='og:novel:book_name']",
                "attribute": "content"
            },
            "content": {
                "selector": "#chaptercontent"
            },
            "titles": {
                "mainTitle": "h1",
                "subTitle": "h2.chapter-title"
            },
            "nextPage": {
                "selector": "a.next",
                "attribute": "href"
            },
            "watermarks": [
                "example\\.com",
                "广告词"
            ],
            "encoding": "utf-8"
        }
    }
}
```

### 适配器选择流程

```
用户输入 URL
    │
    ▼
遍历已知适配器，调用 adapter.match(url)
    │
    ├── 匹配到 → 使用该适配器
    │
    └── 未匹配 → 使用 AutoDetectAdapter
                    │
                    ▼
                抓取页面 HTML
                    │
                    ▼
                依次执行探测策略
                    │
                    ├── 全部成功 → 输出探测结果，开始爬取
                    │              （同时提示用户可将结果保存为配置）
                    │
                    └── 部分失败 → 输出已探测到的信息
                                   提示用户补充 site-config.json
```

### 探测结果缓存

为避免每章都重新探测，首次探测成功后将结果缓存：

```javascript
// 缓存结构
{
    "https://www.example.com": {
        "adapter": "auto",
        "contentSelector": "#chaptercontent",
        "mainTitleSelector": "h1",
        "subTitleSelector": "h2.title",
        "nextPageSelector": "a.next",
        "novelName": "吞噬星空",
        "watermarks": ["example.com"],
        "encoding": "utf-8"
    }
}
```

缓存保存到 `.adapter-cache.json`（已加入 .gitignore）。

### clean.js 改造

clean.js 需要同步改造，支持多站点清洗规则：

```
当前：硬编码 hetushu 水印正则
改造后：
  1. 读取 .adapter-cache.json 获取对应站点的水印规则
  2. 若缓存不存在，使用通用清洗规则（URL、常见广告词）
  3. 支持 site-config.json 中定义的自定义水印
```

### 命令行接口

```bash
# 爬取（传入起始章节 URL）
yarn start https://www.hetushu.com/book/10319/7209353.html

# 爬取 + 自动清洗
yarn start https://www.example.com/chapter/123 --clean

# 仅清洗
yarn clean

# 探测模式（只分析页面结构，不爬取，用于调试）
yarn start https://www.example.com/chapter/123 --detect

# 指定配置文件
yarn start https://www.example.com/chapter/123 --config my-site.json
```

### 索引文件与校验机制

爬取完成后自动生成 `index.json`，记录卷顺序、章节顺序和文件完整性校验信息。

#### index.json 结构

```json
{
    "novelName": "吞噬星空",
    "startUrl": "https://www.hetushu.com/book/10319/7209353.html",
    "crawlDate": "2026-06-17T12:00:00.000Z",
    "totalChapters": 172,
    "volumes": [
        {
            "order": 1,
            "name": "第七卷 九河圣界",
            "chapterCount": 49,
            "chapters": [
                {
                    "index": 1,
                    "fileName": "0001_第九章 归来.html",
                    "url": "https://www.hetushu.com/book/10319/7209353.html",
                    "contentHash": "a1b2c3d4e5f6..."
                }
            ]
        }
    ]
}
```

#### 卷顺序记录

- 爬取过程中维护 `volumeOrder` 数组，新卷名首次出现时按顺序注册
- `index.json` 中 `volumes` 数组严格按首次出现顺序排列
- 保证与网页上的阅读顺序一致（从起始章节开始，按翻页顺序自然排列）

#### 文件完整性校验

- 每个章节保存时计算 HTML 内容的 MD5 哈希，记录在 `contentHash` 字段
- 爬取完成后自动执行 `validateFiles()`，逐文件比对哈希值
- 输出校验结果：通过数 / 失败数，失败文件列出期望哈希和实际哈希（或"文件不存在"）

#### clean.js 适配

- 优先读取 `index.json`，按索引中的卷顺序和章节顺序处理
- 无索引时回退到递归查找文件系统
- 输出目录结构与 HTML 目录结构保持一致

### 实施步骤

**阶段一：基础重构**
1. 抽取 `SiteAdapter` 基类
2. 将当前 `crawler.js` 中 hetushu 的逻辑迁移到 `HetushuAdapter`
3. `crawler.js` 改为通过适配器接口调用

**阶段二：自动探测**
4. 实现 `AutoDetectAdapter` 的五层探测策略
5. 实现探测结果缓存机制
6. 添加 `--detect` 调试模式

**阶段三：配置系统**
7. 实现 `site-config.json` 配置加载
8. 适配器选择流程：配置 > 已知适配器 > 自动探测
9. 更新命令行参数解析

**阶段四：清洗适配**
10. `clean.js` 读取适配器缓存中的水印规则
11. 通用清洗规则兜底

**阶段五：更多站点**
12. 根据实际使用添加常用站点适配器（笔趣阁、69书吧等）

### 风险与注意事项

1. **自动探测准确率**：无法保证 100% 覆盖所有站点，需保留手动配置兜底
2. **翻页探测**：是最难自动化的部分，部分站点使用 JS 动态加载下一页
3. **编码问题**：部分老站点使用 GBK 编码，需依赖 iconv-lite 正确检测
4. **反爬差异**：不同站点反爬策略不同，FlareSolverr 不一定全部适用
5. **性能**：自动探测需要额外请求和分析，首次探测会增加约 1-2 秒开销
